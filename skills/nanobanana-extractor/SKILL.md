# Nanobanana Style Extractor

从图片中逆向工程提取可复用的风格提示词，使用户能够生成风格完全一致的新图片。

## 快速参考

### 核心原则

| 概念 | 说明 |
|------|------|
| **共性元素** | 风格核心，必须保持不变，确保风格一致性 |
| **个性化元素** | 用 `{}` 框起来，可由用户修改或替换 |
| **判断标准** | 改变该元素是否会影响整体风格？是→共性，否→个性化 |

### 输出格式

固定使用罗马数字分层结构：

```
I. Basic Settings (基础设定)
II. Style Core - Keep Unchanged (共性元素)
III. Customizable Elements (个性化元素，用{}标注)
IV. Technical Specifications (技术规格)
V. Mood & Atmosphere (氛围情绪)
Negative Prompts: [排除元素]
```

### Nanobanana 黄金法则

1. **自然语言优先**：使用完整句子，避免 tag soup 格式
2. **具体性至上**：精确描述替代模糊词汇
3. **情境框架**：提供 "for [purpose]" 帮助推断细节
4. **材质纹理**：加入表面质感描述

## 工作流程

### 阶段 1: 图片接收与配置确认

**步骤 1.1** 接收用户上传的图片

**步骤 1.2** 领域自动识别

执行6域分类，输出：
```json
{
  "domain": "photo_portrait",
  "confidence": 85,
  "keywords": ["face", "lighting", "bokeh"],
  "secondary": "painting_realistic"
}
```

如置信度 < 70%，询问用户确认领域。

**步骤 1.3** 存储路径确认（持久化配置）

读取 `~/.claude/nanobanana-config.json` 中的 `style_library_path` 字段：
- **已配置** → 直接使用，无需询问用户
- **未配置/文件不存在** → 询问用户存储路径，保存到配置文件的 `style_library_path` 字段

> 此路径与 nanobanana-illustrator 共享同一个配置文件，确保风格库路径一致。

**步骤 1.4** 询问配置（每次使用都问，仅 2 个问题）

使用 AskUserQuestion 一次性收集：

1. **尺寸比例**：方形(1:1) / 横版(16:9) / 竖版(9:16) / 保持原图比例
2. **关联写作风格**：此视觉风格对应哪个写作风格/账号名？（如"余北北"、"反向的猫"等）

> **为什么要关联写作风格？**
> 文件命名会带上写作风格名前缀（如 `3012-01-反向的猫风格-波普表现主义油画人物.md`），
> 这样 nanobanana-illustrator 在为文章配图时，可以通过文件名自动匹配写作风格对应的视觉风格，
> 无需每次手动选择。

---

### 阶段 2: 风格深度分析

**步骤 2.1** 根据领域读取分析维度

参考 `references/domain-analysis.md` 获取对应领域的：
- 共性元素检查清单（含所有子维度表）
- 个性化元素示例
- 关键维度权重

**步骤 2.1.5** 深度观察（在读取维度后、提取元素前执行）

先不急着填模板，对图片进行逐层观察：

1. **材质清点**：画面中出现了哪些材质？（皮肤、水、头发、布料、金属……）
2. **环境介质判断**：画面的传播介质是空气还是其他？（水下？雾中？烟中？正常空气？）
3. **光线追踪**：光从哪里来？穿过了什么？照在了什么上面？产生了什么效果？
4. **元素交互识别**：哪些视觉效果是由多个元素交互产生的？（如水中焦散光打在皮肤上）

> 这一步的结果将决定后续模板中哪些章节需要填写、哪些可以跳过。
> 例如：如果环境介质是水下，则 Environment Medium 和 Element Interactions 必须详细填写。

**步骤 2.2** 执行元素提取

> **⚠️ 视觉忠实原则（严格执行）**
> - **只描述图片中实际可见的视觉元素**，绝不推测、脑补、或添加图片中不存在的内容
> - **禁止推断用途或场景**：不要因为画面温暖就加"居家"，不要因为有植物就加"生活场景"，不要因为色调柔和就加"ins风"
> - **共性元素 = 可验证的视觉特征**：笔触方向、颜料厚度、色彩关系、光影对比度 — 这些是你能从像素中直接观察到的
> - **个性化元素 = 图片中可见的具体内容**：人物、物体、场景 — 用 `{}` 标注，但仍然必须基于图片实际内容
> - **如果某个维度在图片中无法判断，跳过它**，不要填默认值

对每个共性元素附加复用评分 (1-10)：
- **通用性** (1-10)：是否适用于同类型多数图片
- **清晰度** (1-10)：描述是否足够具体
- **独立性** (1-10)：是否可独立于其他元素复用

综合评分 = (通用性 × 0.4 + 清晰度 × 0.3 + 独立性 × 0.3)

**步骤 2.3** 提取规律公式

不只列出元素，同时提取比例/关系公式：
- 配色规律：`主色60% + 辅色30% + 点缀10%`
- 光影比例：`高光区40% + 中间调50% + 阴影10%`

**步骤 2.4** 分析结论展示与用户确认（必须执行，不可跳过）

将深度分析结论以结构化方式展示给用户，包括：
- 领域分类结果
- 核心技法判断（如笔触类型、渲染方式）
- 关键配色与光影特征
- 共性 vs 个性化的初步划分

然后**必须**使用 AskUserQuestion 询问用户：
> "以上是我的分析结论，你有没有补充或不同建议？"

- **用户有补充/修正** → 根据反馈重新调整分析，再次展示修正后的结论
- **用户确认无补充** → 进入阶段 3 生成提示词

> ⚠️ 此步骤是强制门控：**未经用户确认，禁止进入阶段 3**。
> 这可以避免在错误的分析方向上浪费后续工作。

### 阶段 3: 提示词生成

**步骤 3.1** 应用领域模板

参考 `references/output-templates.md` 选择对应领域的输出模板。

**步骤 3.2** 生成负面提示词

根据领域特征，列出需要排除的常见偏差方向。

---

### 阶段 4: 质量检查

根据领域执行对应检查清单（见 `references/domain-analysis.md`）。

核心标准：
- 共性元素复用评分 ≥ 7 的数量 ≥ 60%
- 用户替换 `{}` 后风格是否能保持一致
- **反幻觉检查**：逐条审查所有提取的元素，删除任何无法从图片像素中直接验证的描述（如推测的使用场景、臆想的环境氛围、与画面无关的风格标签）
- **笔触精度检查**（绘画类）：对比图片实际笔触纹理，确认描述是否准确区分了细腻/粗犷/厚涂/薄涂/干笔/湿润等特征，拒绝一刀切的笼统描述

---

### 阶段 4.5: AI 测试图验证

**步骤 4.5.1** 填充默认变量

- 读取 `III. Customizable Elements` 中所有 `{}` 占位符
- 为每个占位符填入合理的通用默认值（如 `{gender}` → "young woman"、`{scene}` → "daily life moment"）
- 目的：生成一张能体现风格核心特征的通用测试图

**步骤 4.5.2** 组合测试提示词

- 将 Style Core（共性元素）+ 填充后的可定制变量 + Mood & Atmosphere + Negative Prompts 组合成完整提示词
- 在提示词前加引导语："Generate an illustration in this exact visual style:"

**步骤 4.5.3** 调用 API 生成测试图

- 读取 `~/.claude/nanobanana-config.json` 获取 `api_key`、`api_endpoint`、`model`
- 同时将用户上传的原图作为参考图传入（`inline_data`），提升风格还原度
- API 请求格式：

```json
{
  "contents": [{"parts": [
    {"text": "完整测试提示词"},
    {"inline_data": {"mime_type": "image/png", "data": "原图base64"}}
  ]}],
  "generationConfig": {
    "responseModalities": ["IMAGE"],
    "imageConfig": {"aspectRatio": "阶段1.4用户选的比例"}
  }
}
```

- 图片保存路径：`config.image_output_dir`
- 命名：`[风格描述]-测试图.png`（先写 `/tmp` 再 `cp` 到 iCloud）

**步骤 4.5.4** 写入风格文件

- 在风格 `.md` 文件的 `## 原图参考` 之后，新增：

```markdown
## AI 测试图
![[风格描述-测试图.png]]
```

### 阶段 5: 输出与存储

**步骤 5.1** 输出纯提示词（无解释）

**步骤 5.2** 确定文件序号（固定序号，非顺序递增）

序号根据关联写作风格类型决定：
- **有指定写作风格**（如"肥猫说财"、"反向的猫"等）→ 固定使用序号 `01`
- **无指定风格 / 通用** → 固定使用序号 `02`

> 同一序号下可存在多个不同视觉风格文件（通过视觉风格描述区分），
> 例如 `3012-01-反向的猫风格-波普表现主义漫画面板插画.md` 和 `3012-01-反向的猫风格-水彩淡雅人物.md` 可以共存。

**步骤 5.3** 创建存储文件

- 路径：配置文件中的 `style_library_path`
- 命名：`3012-[01或02]-[写作风格名]风格-[视觉风格描述].md`
- 内容：原图引用 + 提示词 + 元数据

**步骤 5.4** 迭代优化（替换原来的简单交付确认）

展示给用户：原图参考 + AI 测试图 + 提示词全文

使用 AskUserQuestion 询问：
> "AI 测试图和原图风格是否一致？"

选项：
- 满意，保存当前版本
- 需要优化（请描述问题）

**→ 用户选择「满意」：**
- 保存最终版本，进入步骤 5.5 完成

**→ 用户选择「需要优化」：**

1. 接收用户的文字反馈（如"颜色太饱和"、"线条应该更粗"、"背景太复杂"）
2. 根据反馈定位并修改提示词中的对应部分：
   - 颜色相关 → 修改 Color Palette / Color Tone
   - 线条相关 → 修改 Line Art
   - 构图相关 → 修改 Composition
   - 氛围相关 → 修改 Mood & Atmosphere
   - 排除项 → 修改 Negative Prompts
3. 展示修改前后的 diff（让用户知道改了什么）
4. 重新调用 API 生成新测试图（覆盖旧测试图文件）
5. 再次展示：原图 + 新测试图 + 更新后的提示词
6. 再次询问是否满意
7. 最多迭代 5 轮，超过后提示用户保存当前最佳版本

**→ 最终保存：**
- 更新风格 `.md` 文件中的提示词和 AI 测试图
- 记录迭代次数到元数据

**步骤 5.5** 交付完成

```
提示词生成完成！已保存至：[文件路径]
迭代次数：[N] 轮

如需后续修改：请直接说明
```

## 6域分类系统

| 域 | 英文标识 | 说明 | 典型示例 |
|---|---|---|---|
| 摄影-人像 | photo_portrait | 以人为主体的照片 | 肖像、时尚、街拍人像 |
| 摄影-场景 | photo_scene | 非人物主体的照片 | 风景、产品、建筑、室内、美食 |
| 绘画-写实 | painting_realistic | 追求真实感的画作 | 油画、厚涂、概念艺术 |
| 绘画-风格化 | painting_stylized | 有明显风格特征的画作 | 动漫、水彩、像素、扁平插画 |
| 平面设计 | graphic_design | 排版/图文结合 | 海报、封面、UI、Logo |
| 通用 | general | 无法明确归类 | 表情包、数据可视化、截图 |

### 分类优先级

1. 首先判断媒介：摄影 vs 绘画 vs 设计
2. 摄影：人物主体 → photo_portrait，否则 → photo_scene
3. 绘画：追求真实感 → painting_realistic，风格明显 → painting_stylized
4. 有明显排版/图文 → graphic_design
5. 无法归类 → general

详细分析维度见 `references/domain-analysis.md`。

## 元素评分机制

### 复用评分标准

| 分数 | 通用性 | 清晰度 | 独立性 |
|------|--------|--------|--------|
| 9-10 | 适用于90%+同类图片 | 可直接复制使用 | 完全独立 |
| 7-8 | 适用于70%+同类图片 | 需少量调整 | 基本独立 |
| 5-6 | 适用于50%+同类图片 | 需要补充说明 | 部分依赖 |
| 1-4 | 仅适用于特定图片 | 描述模糊 | 强依赖其他元素 |

### 评分示例

```
色温: 暖调 3200K [复用评分: 9/10 - 高通用性，可直接复用]
构图: 三分法，主体位于右侧交点 [复用评分: 8/10 - 通用但位置可变]
服装: 白色连衣裙 [复用评分: 3/10 - 个性化内容，应标记为 {}]
```

## 文件存储

### 命名规则

遵循 file-organizer 5层命名规范，并携带写作风格关联：

- 格式：`3012-[01或02]-[写作风格名]风格-[视觉风格描述].md`
- 序号规则（固定，非递增）：
  - `01` = 有指定写作风格关联（如"反向的猫"、"肥猫说财"等）
  - `02` = 无特定来源 / 通用风格
- 写作风格名：来自阶段 1.4 用户指定的关联写作风格，无特定来源用 `通用`
- 视觉风格描述：AI 根据图片风格特征生成的简短中文描述

**命名示例：**
```
3012-01-反向的猫风格-波普表现主义漫画面板插画.md   ← 指定风格，序号01
3012-01-肥猫说财风格-压抹水彩动态人物.md           ← 指定风格，序号01
3012-02-通用风格-极简蓝色线描人物.md               ← 通用风格，序号02
3012-02-通用风格-科技暗色封面人物抠图.md           ← 通用风格，序号02
```

> **下游联动**：nanobanana-illustrator 通过扫描文件名中的 `[写作风格名]风格` 前缀，
> 自动匹配当前写作风格对应的视觉提示词，实现无感配图。


## 异常处理

### 图片模糊或信息不足
说明问题，询问是否可提供更清晰图片或补充描述。

### 领域判断置信度低
展示分类结果，询问用户按哪个领域分析。

### 风格元素过于独特
尽可能用已有术语组合描述，补充具体视觉特征，在负面提示词中排除可能偏差。

## 迭代改进

### 改进方向

1. **共性元素识别准确性**：用户反馈替换 `{}` 后风格是否一致
2. **领域分类准确性**：记录分类错误的案例，优化分类规则
3. **复用评分校准**：根据实际复用效果调整评分标准

### 反馈收集

当用户回复 `#完成` 时：
- 记录本次提取的领域类型
- 记录复用评分分布
- 记录用户修改反馈（如有）
