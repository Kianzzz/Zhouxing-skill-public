# Web OpenCLI

通过 OpenCLI 浏览器自动化技术进行网络搜索。复用浏览器登录态，无需手动管理 Cookie，支持 7 个精选平台的深度内容搜索。

## 核心优势

- **零凭证管理**：复用 Chrome/Chromium 已登录状态，无需手动配置 Cookie
- **反检测能力**：内置多层反检测机制，不易被封禁
- **深度内容访问**：可访问登录后才能看到的完整内容
- **稳定可靠**：基于真实浏览器环境，行为模式接近人类

## 支持平台

**国内平台**（2 个）：
- 微博：实时热点、社会讨论（二手+三手）
- 小红书：用户体验、生活化分享（二手）

**国外平台**（3 个）：
- Twitter：实时动态、开发者一手信息（⭐ 一手源）
- Reddit：真实用户讨论、踩坑经验（⭐ 一手+二手）
- YouTube：视频教程、演示内容（二手）

**为什么只有 5 个平台？**
- 专注一手和二手信息源，保证内容质量
- 移除了抖音（三手内容为主、短视频信息密度低）
- 移除了微信公众号（只能搜索特定账号，更适合用 benchmark-monitor skill）

## 工作流程

### 步骤 0a：用户画像建档（首次使用）

首次使用时，通过 AskUserQuestion 采集用户基础信息，存储到 `~/.web-opencli/user-profile.json`：

**采集内容**：
1. **你的身份角色？** — 自媒体创作者 / 独立开发者 / 产品经理 / 研究者 / 学生 / 其他
2. **你搜索时最看重什么信息类型？** — 一手信息源(开发者推文/官方发布) / 深度分析(评测/教程) / 案例模板 / 趋势热点
3. **你的目标受众？** — 普通用户 / 技术人员 / 行业从业者 / 自己学习用

**存储结构**：
```json
{
  "identity": "AI自媒体创作者",
  "info_preference": "一手信息源",
  "target_audience": "普通用户",
  "created": "2026-04-04",
  "platform_weights": {
    "twitter": 1.0,
    "reddit": 1.0,
    "youtube": 1.0,
    "xiaohongshu": 1.0,
    "weibo": 1.0
  },
  "topic_domains": {}
}
```

**画像应用**：
- `info_preference = "一手信息源"` → 提高 Twitter 权重，降低微博权重（三手内容多）
- `target_audience = "普通用户"` → 提高小红书/YouTube 权重（内容更易懂）
- `identity` + 搜索话题 → 个性化解读意图（如"AI自媒体创作者"搜"写作技巧" → 偏向"AI辅助写作"）

**动态更新**：
- 每次搜索后，根据用户反馈自动调整 `platform_weights`
- 记录用户常搜的话题领域到 `topic_domains`

### 步骤 0b：环境检测（首次使用）

首次使用时自动检测 OpenCLI 环境：

1. **检查 OpenCLI 安装**
   ```bash
   which opencli
   ```
   
2. **检查守护进程和扩展状态**
   ```bash
   opencli doctor
   ```

3. **如果未安装或未配置**，引导用户完成配置：
   - 安装 OpenCLI：`npm install -g @jackwener/opencli`
   - 下载浏览器扩展：https://github.com/jackwener/opencli/releases
   - 安装扩展到 Chrome/Chromium
   - 在浏览器中登录目标网站（知乎、小红书、微博等）
   - 验证连接：`opencli doctor`

**环境检测脚本**：使用 `scripts/check_opencli.sh` 自动检测并给出修复建议。

### 步骤 1：意图识别

分析用户搜索意图，判断意图类型：

| 意图类型 | 特征词 | 用户真正想要 |
|---------|--------|-------------|
| **资讯动态** | "最新"、"发布"、"更新"、"新闻"、"最近" | 最近发生的事件和发布（一手新闻） |
| **教程指南** | "教程"、"怎么做"、"步骤"、"入门"、"配置" | 从零开始的操作指南（可复现） |
| **实战案例** | "案例"、"例子"、"怎么用"、"实战"、"赚钱" | 真实使用案例、数据结果（一手干货） |
| **观点评价** | "评价"、"怎么样"、"值不值"、"对比" | 用户评价、产品对比、优缺点 |
| **深度内容** | "深度"、"详细"、"完整"、"架构"、"原理" | 深度分析、技术解析（原创干货） |

**意图优先级**：
- 如果用户明确提到时间（"最近3天"、"今天"），优先识别为"资讯动态"
- 如果用户问"怎么做"、"如何配置"，优先识别为"教程指南"
- 如果用户问"有什么案例"、"别人怎么用"，优先识别为"实战案例"

**歧义判断**：如果搜索词过于宽泛或存在多种理解，用 AskUserQuestion 澄清意图。

### 步骤 2：关键词生成与扩展

基于意图生成结构化关键词：

**核心词**（1-2 个）：直接描述用户需求
**扩展词**（2-4 个）：同义词、相关概念、上下游词汇
**生态词**（可选）：竞品、上下游工具、相关事件

**关键词生成规则**：
- 中英文双语（每个核心词同时生成中英文版本）
- 避免歧义词（排除已明确不需要的方向）
- 加入场景限定词（如"实战"、"案例"、"tutorial"）

**智能关键词扩展**（当搜索词宽泛时）：

如果用户搜索宽泛的话题（如"openclaw AI 信息"），应扩展到相关生态：

```
用户搜索："openclaw AI 信息"

关键词扩展：
├─ 核心词：openclaw
├─ 竞品词：hermes agent, manus, cursor（AI Agent 生态）
├─ 上游词：claude, anthropic, 模型更新（依赖的基础设施）
├─ 下游词：clawhub, 技能市场, 应用案例（生态应用）
└─ 事件词：政策变化, 订阅调整, 安全事件（影响使用的事件）
```

**扩展判断标准**：
- 用户搜"openclaw AI 信息"（宽泛）→ 扩展生态词
- 用户搜"openclaw 安装教程"（精确）→ 不扩展
- 用户搜"AI Agent 工具"（通用）→ 扩展到 openclaw、hermes、manus 等

### 步骤 3：平台选择与动态权重

根据意图和话题特征选择搜索平台，并应用动态权重。

**话题类型与平台权重**：

| 话题类型 | 平台权重（从高到低） | 原因 |
|---------|---------------------|------|
| **技术前沿/产品发布** | Twitter(1.5x) > Reddit(1.3x) > YouTube(1.2x) > 小红书(1.0x) > 微博(1.0x) | 海外开发者一手信息多 |
| **实战经验/案例分享** | Twitter(1.4x) > YouTube(1.4x) > Reddit(1.3x) > 小红书(1.2x) > 微博(1.0x) | Twitter 有高质量实战帖，YouTube 有完整演示 |
| **本地化场景（中国特色）** | 小红书(1.5x) > 微博(1.3x) > Twitter(1.0x) > Reddit(0.8x) > YouTube(0.8x) | 连接微信、国内支付、中文场景优化 |
| **教程指南** | YouTube(1.5x) > 小红书(1.3x) > Reddit(1.2x) > Twitter(1.0x) > 微博(1.0x) | 视频教程质量高，小红书有保姆级图文 |
| **实时热点** | Twitter(1.5x) > 微博(1.5x) > Reddit(1.2x) > 小红书(1.0x) > YouTube(0.8x) | Twitter 和微博实时性强 |

**平台选择规则**：
- 每次搜索至少覆盖 **3 个平台**
- 根据话题类型选择权重最高的 3-5 个平台
- 中文话题：至少 1 个国内 + 2 个国外
- 英文话题：至少 2 个国外 + 1 个国内
- 跨语言话题：至少 2 个国内 + 2 个国外

**多样性约束**：
- 最终结果必须覆盖至少 2 个海外平台 + 2 个国内平台
- 单一平台最多占最终结果的 **40%**（而不是 50%）
- 确保国内优质内容不会因权重低而被过滤

### 步骤 4：并行搜索

**所有平台搜索必须并行执行**——在单次响应中同时发起多个 OpenCLI 命令。

**搜索前读取平台经验**：
- 检查 `~/.web-opencli/platform-experience/{platform}.json` 是否存在
- 如果存在，读取该平台的历史搜索经验：
  - 成功的关键词组合
  - 有效的搜索策略
  - 已知的登录检测特征
  - 需要避免的陷阱

**搜索命令格式**：

```bash
# 微博
opencli weibo search "关键词" --limit 20 --format json

# 小红书
opencli xiaohongshu search "关键词" --limit 20 --format json

# Twitter
opencli twitter search "keyword" --limit 20 --format json

# Reddit
opencli reddit search "keyword" --limit 20 --format json

# YouTube
opencli youtube search "keyword" --limit 20 --format json
```

**搜索量分配**：
- 主力平台：20-25 条
- 辅助平台：10-15 条
- 总量控制在 80-120 条

**失败兜底**：
- 如果 OpenCLI 命令失败，记录错误信息和失败原因到平台经验文件
- 在最终输出的筛选说明中标注失败平台
- 不使用 WebSearch 兜底（与 web-search skill 职责分离）

**搜索后沉淀经验**：
- 记录本次搜索的关键词组合和结果质量
- 如果某个关键词在某平台产出了高质量结果（评分 ≥ 85），记录到该平台经验文件
- 如果遇到登录墙或反爬检测，记录触发条件

### 步骤 5：结果解析

OpenCLI 返回 JSON 格式结果，提取关键字段：

**通用字段映射**：
```json
{
  "title": "标题",
  "author": "作者",
  "url": "链接",
  "published_at": "发布时间",
  "engagement": {
    "likes": "点赞数",
    "comments": "评论数",
    "shares": "分享数"
  },
  "excerpt": "摘要"
}
```

**平台特殊字段**：
- Twitter: `retweets`, `likes`
- Reddit: `score`, `num_comments`
- 小红书: `likes`, `collects`
- 微博: `reposts`, `comments`, `likes`
- YouTube: `views`, `likes`

### 步骤 6：内容类型分类与动态评分

#### 6.1 内容类型分类（基础分类）

首先对每条结果进行内容类型分类：

| 内容类型 | 特征 | 判断标准 |
|---------|------|---------|
| **一手新闻** | 官方公告、产品发布、事件通知 | 来自官方账号；包含"宣布"、"发布"、"推出"；信息价值高但可能无实操细节 |
| **一手干货** | 开发者实战分享、技术深度解析、完整案例复盘 | 有"我"、"我们"、"我的"；有数据、有过程、有结果；既是原创又有实操价值 |
| **深度教程** | 保姆级教程、代码示例、视频演示 | 有明确步骤（1/2/3...）；有代码/配置示例；有截图/演示；可复现 |
| **经验讨论** | 思路分享、问题讨论、简短经验 | 有思路但无具体方法；纯观点/评价；有结论但无过程 |
| **新闻转发** | 二手/三手信息 | 简单转发、无信息增量、标题党 |

**判断优先级**：
1. 先判断是否是"新闻转发"（如果是，直接降权）
2. 再判断是否有实操价值（步骤/代码/数据）→ 深度教程 or 一手干货
3. 判断是否是官方/原创者 → 一手新闻 or 一手干货
4. 其他归为"经验讨论"

#### 6.2 根据搜索意图动态调整权重

根据用户的搜索意图，动态调整内容类型的权重：

| 搜索意图 | 内容类型权重（加分） | 原因 |
|---------|---------------------|------|
| **资讯动态** | 一手新闻(+30) > 一手干货(+25) > 经验讨论(+15) > 深度教程(+10) | 用户想知道"发生了什么" |
| **教程指南** | 深度教程(+30) > 一手干货(+25) > 经验讨论(+15) > 一手新闻(+5) | 用户想知道"怎么做" |
| **实战案例** | 一手干货(+30) > 深度教程(+20) > 经验讨论(+15) > 一手新闻(+5) | 用户想知道"别人怎么用的" |
| **观点评价** | 经验讨论(+25) > 一手干货(+25) > 深度教程(+15) > 一手新闻(+10) | 用户想知道"大家怎么看" |
| **深度内容** | 一手干货(+30) > 深度教程(+25) > 经验讨论(+10) > 一手新闻(+5) | 用户想要深度理解 |

#### 6.3 完整评分公式

```
最终得分 = (基础分 + 意图匹配分) × 平台权重 + 时效性分 + 流量分

其中：
- 基础分：60-70 分（内容相关度、完整性）
- 意图匹配分：根据内容类型和搜索意图，+5 到 +30 分
- 平台权重：根据话题类型，0.8x 到 1.5x
- 时效性分：
  - 用户要求"最近3天"：3天内(+30) > 3-7天(+15) > 7-30天(+5) > 30天以上(0)
  - 用户要求"最近7天"：7天内(+30) > 7-30天(+15) > 30天以上(0)
  - 无时间要求：30天内(+20) > 3月内(+10) > 更早(0)
- 流量分：根据平台中位数，+0 到 +20 分
```

**评分示例**：

```
用户搜索："最近3天 openclaw AI 信息"（意图：资讯动态）

内容 A：Anthropic 官方公告（Twitter，3天内）
- 基础分：65
- 意图匹配：+30（一手新闻，完美匹配"资讯动态"）
- 平台权重：×1.5（Twitter 在"技术前沿"权重高）
- 时效性：+30（3天内）
- 流量：+15（60万 views）
→ 最终得分：(65+30)×1.5+30+15 = 187.5

内容 B：小红书深度教程（30天前）
- 基础分：70
- 意图匹配：+10（深度教程，不太符合"资讯动态"）
- 平台权重：×1.0（小红书在"技术前沿"权重一般）
- 时效性：0（30天前，不符合"最近3天"）
- 流量：+10（1000 likes）
→ 最终得分：(70+10)×1.0+0+10 = 90

结果：内容 A 排在前面
```

#### 6.4 多样性约束

- 单一平台最多占最终结果的 **40%**
- 最终结果必须覆盖至少 **2 个海外平台 + 2 个国内平台**
- 内容质量优先：国内优质内容（一手干货、深度教程）不会因平台权重低而被过滤

### 步骤 7：条目式输出

**实现方式**：

使用 Claude 的文本输出能力，直接在对话中输出 Markdown 格式的搜索结果。不需要写入文件，不需要调用外部工具。

**输出流程**：
1. 将评分后的结果按分数排序（降序）
2. 取前 10-15 条
3. 为每条结果生成摘要（基于 `title` 和 `excerpt` 字段）
4. 识别质量亮点（高流量、一手源、权威作者等）
5. 按模板格式化输出到对话中

**输出后收集反馈**：
- 用 AskUserQuestion 询问用户对本次搜索结果的评价：
  - A) 很好，正是我要的
  - B) 还行，但部分结果无关
  - C) 不好，大部分不是我想要的
- 如果选 B/C，追问"哪些类型的结果你不需要？"或"哪个平台的结果质量不好？"
- 将反馈写入 `~/.web-opencli/user-profile.json` 的 `feedback_history` 数组
- 根据反馈自动调整 `platform_weights`：
  - 多次否定某平台 → 该平台权重 -0.1
  - 多次肯定某平台 → 该平台权重 +0.1
  - 多次反映"新闻太多" → 降低微博权重

**输出格式**：

```markdown
📅 2026-04-04

🔍 搜索关键词：${keyword}
🎯 识别意图：${intent_type} — ${intent_description}
📊 原始结果：${total_count} 条 → 筛选后：${filtered_count} 条

[一段话总结]（2-3句话概括整体发现）

---

### 1. ${title}
**平台**: ${platform} | **评分**: ${score}/100
**流量**: ${engagement_metrics} | **发布**: ${date}
**摘要**: ${summary}
**链接**: ${url}
**质量亮点**: ${quality_highlights}

---

📌 筛选说明：
- 意图识别：${intent_type}
- 搜索平台：微博 ${n}条, 小红书 ${n}条, Twitter ${n}条, Reddit ${n}条, YouTube ${n}条
- 筛选标准：动态权重评分 + 信息源层级
- 失败平台：${failed_platforms}（如有）
```

**摘要质量约束**：
- ❌ 禁止空洞开头（不得使用"本文介绍了…"等模板式开头）
- ✅ 保留具体信息（人名、数字、日期、版本号等必须原样保留）
- ✅ 字数限制：50-150 字
- ✅ 信息密度：让读者无需点开链接就能判断"这条对我有没有用"

**质量亮点识别**：

自动识别并标注每条结果的质量亮点：
- 🔥 高流量（点赞/转发/评论数超过平台中位数 2 倍）
- ⭐ 一手源（官方公告、开发者原创）
- 💎 一手干货（有数据、有过程、有结果的实战分享）
- 📚 深度教程（有步骤、有代码、可复现）
- 👤 权威作者（官方账号、领域专家）
- 🆕 最新内容（符合用户时间要求）

**深度内容判断标准**（更严格）：
- ✅ 有明确步骤（第一步、第二步...）
- ✅ 有代码/配置示例（可复制粘贴）
- ✅ 有截图/演示（可验证）
- ✅ 有数据结果（花了多少钱、节省多少时间）
- ❌ 只有观点/思路，无具体方法 → 不是深度内容

**输出示例**：

```markdown
📅 2026-04-04

🔍 搜索关键词：AI 写作工具
🎯 识别意图：实战案例 — 用户想了解真实使用体验和工具推荐
📊 原始结果：85 条 → 筛选后：12 条
```

## 与 web-search 的关系

**职责分工**：

| Skill | 负责平台 | 技术栈 | 适用场景 |
|-------|---------|--------|---------|

| **web-opencli** | 微博/小红书/Twitter/Reddit/YouTube | OpenCLI 浏览器自动化 | 需登录、一手/二手信息源 |

**互补使用**：
- 用 web-opencli 搜索 5 个精选平台（一手/二手信息源）

- 两个 skill 的输出格式一致，便于对比

## 执行规则

### 底线规则

- ❌ 不把搜索结果消化成"指南"——用户需要原始信息源
- ❌ 不跳过意图识别——歧义搜索浪费所有后续资源
- ❌ 不在意图有歧义时自行假设——猜错方向的结果对用户毫无价值
- ❌ 不在 OpenCLI 环境未就绪时强行搜索——先引导用户完成配置
- ❌ 不使用 WebSearch 兜底——与 web-search skill 职责分离

### 搜索流程（7 步）

0. **环境检测**：首次使用检查 OpenCLI 环境，未就绪时引导配置
1. **意图识别**：分析用户输入，判断意图类型（资讯动态/教程指南/实战案例/观点评价/深度内容）；存在歧义时用 AskUserQuestion 澄清
2. **关键词生成与扩展**：基于意图生成核心词 + 扩展词；如果搜索词宽泛，扩展到相关生态（竞品、上下游工具、事件）
3. **平台选择与权重**：根据话题类型选择 3-5 个平台，应用动态权重（技术前沿 → Twitter 权重高；本地化场景 → 小红书权重高）
4. **并行搜索**：**并行执行所有平台搜索**（60-100 条）
5. **结果解析**：提取标准字段，统一数据结构
6. **内容分类与动态评分**：
   - 先对每条结果进行内容类型分类（一手新闻/一手干货/深度教程/经验讨论/新闻转发）
   - 根据搜索意图动态调整内容类型权重
   - 应用平台权重、时效性分、流量分
   - 应用多样性约束（单一平台最多 40%，至少 2 个海外 + 2 个国内）
   - 筛选出 10-15 条
7. **条目输出**：按模板输出，标注质量亮点和失败平台
