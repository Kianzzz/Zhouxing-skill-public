# Nanobanana Illustrator

为文章自动生成配图，基于内容结构智能判断配图位置，支持 AI 风格图生成、Pexels 真实照片下载、TMDB 电影剧照和信息图四种模式，将图片保存并插入 Markdown。

## 快速参考

### 图片来源

| 来源 | 说明 | 适用场景 |
|------|------|---------|
| **风格图** | AI 生成风格化插画 | 需要统一视觉风格的文章 |
| **真实照片** | 从 Pexels 下载免费高清照片 | 需要真实场景/物品/风景的文章（禁止出现人类） |
| **电影剧照** | 从 TMDB 获取电影背景剧照 | 成长/生活/情感类文章，AI 根据情绪自动选片（允许人物） |
| **信息图** | AI 生成结构化信息图 | 干货/方法/数据/对比类内容，10种布局×6种风格，文字精准度优先 |

### 核心模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **封面** | 仅生成1张封面 | 所有文章 |
| **配图** | 仅为文章正文配图 | 内容丰富的文章 |
| **都要** | 封面+正文配图 | 重要/长文章 |

## 初次配置

### ⛔ 门控检查（强制，最高优先级）

**每次启动 skill 时，第一步必须执行此检查，不可跳过：**

```
读取 ~/.claude/nanobanana-config.json
  → 文件不存在？ → 强制进入「初次配置流程」，完成后才能继续
  → 文件存在但缺少 api_key？ → 提示用户补充 API Key
  → 文件存在且完整？ → 跳过配置，进入「预执行参数收集」
```

**⚠️ 硬性约束：在 config.json 完整之前，禁止执行任何配图操作。**

### 初次配置流程

第一次使用 nanobanana-illustrator 时，按顺序配置以下三项：

#### 1. 风格库路径

询问：「风格文件存放在哪里？」

推荐位置：
- `📚 30-积累/301-指令/3012-视觉提示词/` （提示词库）
- `📤 40-输出/402-风格库/` （输出风格库）
- 自定义路径

→ 保存到配置文件，后续自动读取

#### 2. 图片存储目录

询问：「生成的配图存放在哪里？」

推荐位置：
- `📤 40-输出/401-图片/` （统一图片目录）
- `assets/images/` （项目内assets）
- 自定义路径

→ 所有生成的图片都保存到这个目录

#### 3. API 密钥

询问：「Nanobanana API Key？」

- 用户提供 Google Gemini API Key（格式：AIzaSy...）
- 首次需要提供，后续无需重复输入
- 保存到本地安全配置

**获取后必须执行 API 连通性测试：**
```
1. 调用 Gemini API ListModels 验证 Key 有效性
2. 确认 gemini-3-pro-image-preview 模型可用
3. 发送一张简单的测试图片生成请求
4. 测试通过 → 提示"✅ API 配置成功"
5. 测试失败 → 提示错误原因，请用户重新提供 Key
```

## 工作流程

### 阶段 1: 预执行参数收集

启动 skill 时，一次性收集所有参数，避免中途中断。

**⚠️ 单篇 vs 多篇模式判断：**
- 用户提供 1 篇文章 → 进入「单篇模式」（下方流程）
- 用户提供 2+ 篇文章 → 进入「批量模式」（见下方「批量模式流程」）

---

#### 单篇模式

**步骤 1.0** 图片来源选择（最先询问）
```
Q: 配图来源？
→ 风格图（AI 生成风格化插画）
→ 真实照片（从 Pexels 下载高清照片）
→ 电影剧照（从 TMDB 获取电影背景剧照）
→ 信息图（AI 生成结构化信息图，10种布局×6种风格）
```

**步骤 1.1** 任务类型选择
```
Q: 任务类型？
→ 封面 / 配图 / 都要
```

> 注意：真实照片模式下，封面和正文配图均可使用 Pexels 照片。用户选择的图片来源对所有任务类型（封面/配图/都要）均生效，不自动回退。

**步骤 1.2** 文章路径指定
```
Q: 要配图的文章路径？
→ 用户指定 markdown 文件
```

**步骤 1.3** 风格自动匹配（仅风格图模式，真实照片模式和电影剧照模式跳过此步）

从文章文件名自动匹配视觉风格，**不主动询问用户**：

```
1. 从文章文件名提取写作风格名
   文件名格式：YYYYMMDD_[写作风格名]_[标题].md
   示例：20260207_余北北吖_经济下行时期说出去的话都是债.md
   提取：余北北吖

2. 扫描风格库（style_library_path）中所有 3012-*.md 文件
   文件名格式：3012-XX-[风格名]风格-[描述].md
   示例：3012-01-余北北风格-极简蓝色线描人物.md
   提取：余北北

3. 模糊匹配：检查「视觉风格名」是否包含在「写作风格名」中，或反之
   余北北 ∈ 余北北吖 → ✅ 命中

4. 匹配结果处理：
   → 唯一匹配 → 自动使用，在确认汇总中告知用户
   → 多个匹配 → 让用户在匹配项中选择
   → 无匹配   → 回退：列出所有风格文件让用户手动选择
```

> **备用匹配源**：如文件名无法提取，可从文章 frontmatter 的 `作品风格` 字段提取（如 `余北北吖_流水账体` → `余北北吖`）。

**步骤 1.4** 内容主题确认
```
Q: 文章主要类型？
→ 科普 / 教程 / 产品评测 / 旅行 / 故事 / 观点评论 / 新闻 / 其他

（可影响配图数量和主题识别准确度）
```

**步骤 1.4.5** 关键词来源确认（仅真实照片模式，电影剧照模式跳过——全自动无需关键词，**必问不可跳过**）
```
Q: 搜索关键词由谁提供？
→ 我自己提供关键词
→ AI 根据文章内容自动提炼

选择「自己提供」→ 收集用户关键词列表，后续按关键词逐个搜索
选择「AI 提炼」→ 在文章分析阶段自动为每个配图位置生成英文搜索关键词
```

**步骤 1.5** 确认配置

**风格图模式：**
```
显示汇总：
  - 来源：风格图（AI 生成）
  - 任务：[类型]
  - 风格：[风格名称]
  - 文章：[文件路径]
  - 主题：[类型]

Q: 确认无误，开始执行？Y/N
```

**真实照片模式：**
```
显示汇总：
  - 来源：真实照片（Pexels）
  - 任务：[类型]
  - 文章：[文件路径]
  - 主题：[类型]
  - 关键词：[用户提供 / AI 提炼]

Q: 确认无误，开始执行？Y/N
```

**电影剧照模式：**
```
显示汇总：
  - 来源：电影剧照（TMDB）
  - 任务：[类型]
  - 文章：[文件路径]
  - 主题：[类型]
  - 选片方式：AI 根据文章情绪自动推荐

Q: 确认无误，开始执行？Y/N
```

→ 确认后进入 **无中断自动执行** 模式

#### 批量模式（多篇文章）

当用户一次提供多篇文章时，使用批量参数收集流程。

**⚠️ 核心原则：每篇文章的配置独立，不同文章可以使用不同的来源和设置。**

**步骤 B1** 统一参数收集

先收集所有文章共享的参数：
```
Q: 任务类型？（对所有文章统一）
→ 封面 / 配图 / 都要
```

**步骤 B2** 逐篇配置表

为每篇文章确定图片来源和模式特定参数，以表格形式一次性确认：

```
请确认每篇文章的配图设置：

| # | 文章 | 来源 | 风格/关键词 |
|---|------|------|-----------|
| 1 | 余北北吖_经济下行... | 风格图 | 自动匹配：余北北风格 |
| 2 | 反向的猫_为什么... | 真实照片 | 关键词：? |
| 3 | 肥猫说财_基金定投... | 信息图 | AI 分析文章内容 |
| 4 | 余北北吖_断亲之后... | 风格图 | 自动匹配：余北北风格 |
| 5 | 某作者_那些年的遗憾... | 电影剧照 | AI 自动选片 |

需要确认：
1. 每篇的「来源」是否正确？
2. 真实照片的文章：关键词由你提供还是 AI 提炼？
```

**自动推断规则（用户未明确指定时）**：
```
1. 文章文件名能匹配到风格库中的风格文件 → 默认「风格图」
2. 文章文件名无法匹配任何风格文件 → 默认「真实照片」
3. 用户可在确认表中修改任何一篇的来源（包括改为「电影剧照」或「信息图」）
```

**步骤 B3** 真实照片文章的关键词来源（**必问**）

如果批量中有任何文章使用真实照片，**必须**询问关键词来源（电影剧照文章不需要关键词确认，全自动选片）：
```
Q: 使用真实照片的文章（#2 反向的猫, #3 肥猫说财），搜索关键词由谁提供？
→ 全部由 AI 提炼
→ 全部由我提供
→ 逐篇指定（有的 AI 提炼，有的我提供）
```

> **⚠️ 硬性约束：只要批量中有真实照片文章，此步骤不可跳过。即使只有 1 篇用真实照片，也必须确认关键词来源。**

**步骤 B4** 汇总确认

```
📋 批量配图计划：

任务类型：封面

| # | 文章 | 来源 | 配置 |
|---|------|------|------|
| 1 | 余北北吖_经济下行... | 风格图 | 余北北风格-极简蓝色线描 |
| 2 | 反向的猫_为什么... | 真实照片 | 关键词：AI 提炼 |
| 3 | 肥猫说财_基金定投... | 信息图 | AI 分析文章内容 |
| 4 | 余北北吖_断亲之后... | 风格图 | 余北北风格-极简蓝色线描 |

Q: 确认无误，开始执行？Y/N
```

→ 确认后按顺序逐篇执行，每篇走各自模式的完整流程（阶段 2 → 阶段 3 → ...）

### 阶段 2: 文章分析

**步骤 2.1** 解析文章结构

读取 markdown 文件，识别：
- 主标题（H1）
- 分小节标题（H2/H3）
- 段落内容
- 特殊区块（代码、表格、列表）

**步骤 2.2** 提取关键信息

对每个分小节提取：
- 标题文本
- 内容摘要（前100字）
- 核心关键词
- 段落数量和总字数

**步骤 2.3** 判断主题契合度

根据用户指定的「内容主题类型」，评估：
- 该分小节是否适合配图
- 配图的语义关联度

参考 `references/content-analysis.md` 中的主题评分规则。

---

### 阶段 3: 智能配图位置判断

**步骤 3.1** 识别「有配图价值」的分论点

过滤掉纯观点、理论性内容，保留有「可视化价值」的内容：

✅ 适合配图的分小节：
- 具体的步骤/流程
- 现象/物体/场景描述
- 数据/结果演示
- 故事情节/场景

❌ 不适合配图的分小节：
- 纯思想论点
- 理论解释
- 抽象观念
- 结论/总结

**步骤 3.2** 判断每个分小节的配图类型

对每个有配图价值的分小节，判断属于哪种类型：

| 内容特征 | 配图类型 | 主体 | 设计思路 |
|---------|---------|------|---------|
| 案例、故事、观点、情感 | **场景图** | 人物 + 场景 | 从内容提取具体场景，用风格提示词生成氛围匹配的插画 |
| 干货、方法、步骤、高密度信息 | **信息图** | 关键信息 + 知识卡片 | 提取核心要点，做图文并茂的知识卡片，人物/场景作为点缀 |

**场景图示例**：文章讲到「朋友帮忙后对方承诺请吃饭却没兑现」→ 设计场景：两人对坐吃饭、一人看手机等消息
**信息图示例**：文章总结「三件事：承诺要兑现、不拿难处开玩笑、分享好消息看场合」→ 设计知识卡片：三个要点配图标

**步骤 3.3** 确定配图数量

```
配图数 = 有配图价值的分小节数量

特殊情况：
- 如果总共只有1个小节 → 只配封面，不配正文
- 如果有10+个分小节 → 可视化价值筛选，避免过度配图
```

**步骤 3.4** 确定配图位置

对每个「有配图价值」的分小节：
- 位置：该小节内容的中段或末尾
- 位置不均匀：完全基于内容自然分布（不强制均匀）
- 避开特殊区块：不在代码/表格/列表中间插入

参考 `references/positioning-algorithm.md`。

### 阶段 3.5: 场景设计（核心环节）

**这是连接「文章内容」和「风格提示词」的桥梁。不可跳过。**

#### ⚠️ 步骤 3.5.0: 风格-内容调和（最高优先级，在设计任何场景之前执行）

**核心原则：文章提供情绪和主题，风格提供视觉词汇表。场景设计必须在风格的视觉词汇表里找到能表达同样情绪的画面。**

**执行流程：**

```
1. 提取文章的情绪内核
   - 不是提取"发生了什么事"，而是提取"这件事让人感到什么"
   - 示例：过年回家 → 情绪内核 = 团圆、温暖、归属感
   - 示例：朋友爽约 → 情绪内核 = 失落、被辜负、小心翼翼

2. 分析风格提示词的调性 DNA
   读取风格文件，判断其视觉世界：
   - 文化语境：东方/西方/无特定文化/混合
   - 时代感：古典/现代/未来/无特定时代
   - 典型场景：这种风格天然适合什么画面？
     · 西方油画壁画 → 宴会厅、花园、教堂、古典建筑、寓言场景
     · 极简线描插画 → 日常生活、城市街头、室内空间
     · 水彩风景 → 自然风光、季节变换、户外活动
     · 赛博朋克 → 霓虹城市、科技设备、未来空间
   - 视觉禁忌：这种风格里明显不协调的元素
     · 西方油画里的红灯笼、春联
     · 极简线描里的繁复装饰
     · 水墨画里的霓虹灯

3. 兼容性判断（三级）
   将文章的具象场景与风格的视觉世界对照：

   🟢 高兼容：文章场景天然属于风格的视觉世界
      → 直接使用文章的具象场景，用风格渲染即可
      · 示例：咖啡店聊天 + 极简线描 → 直接画咖啡店场景

   🟡 中兼容：场景本身不冲突，但有文化/时代元素需要调整
      → 保留场景骨架，替换不兼容的细节元素
      · 示例：过年聚餐 + 水彩插画 → 保留"聚餐"场景，
        但不画春联/红包，改为温暖色调的家庭聚餐氛围

   🔴 低兼容：文章的具象场景与风格的视觉世界明显冲突
      → 必须做「情绪转译」：提取情绪内核，在风格的视觉世界中
        找到承载同样情绪的全新场景
      · 示例：过年团圆 + 西方油画壁画
        情绪内核 = 团圆、温暖、丰盛
        转译场景 = 文艺复兴式长桌宴会、金色烛光、亲人举杯
      · 示例：被催婚的压力 + 古典油画
        情绪内核 = 被审视、无处可逃
        转译场景 = 宫廷中被众人注视的年轻人、沉重的目光
```

**判断速查表：**

| 文章场景 | 风格类型 | 兼容性 | 处理方式 |
|---------|---------|--------|---------|
| 日常生活 | 现代插画 | 🟢 | 直接使用 |
| 日常生活 | 古典油画 | 🔴 | 情绪转译 |
| 职场故事 | 商务摄影 | 🟢 | 直接使用 |
| 职场故事 | 水墨画 | 🟡 | 调整细节 |
| 中式传统场景 | 西方油画 | 🔴 | 情绪转译 |
| 中式传统场景 | 国风插画 | 🟢 | 直接使用 |
| 情感/关系 | 任何风格 | 🟢/🟡 | 情绪通常可跨风格表达 |

**⚠️ 关键约束：情绪转译时，绝不生造风格世界中不存在的元素。** 如果风格是西方油画，就在西方油画的经典场景库里找对应情绪的画面，而不是把中国元素"画成油画风格"。

#### 封面场景设计

```
1. 提取文章整体主题和核心情绪
2. 执行风格-内容调和（步骤 3.5.0），确定兼容性级别
3. 根据兼容性级别设计场景：
   - 🟢 高兼容：直接基于文章主旨设计视觉场景
   - 🟡 中兼容：基于文章主旨设计，替换不兼容细节
   - 🔴 低兼容：基于情绪内核，在风格的视觉世界中重新构思场景
4. 场景要素：人物状态 + 环境 + 动作 + 情绪氛围
5. 将场景要素映射到风格提示词的可定制变量中
```

**示例（高兼容 🟢）**：
- 文章主题：「经济下行时期，说出去的话都是债」
- 风格：极简蓝色线描人物
- 兼容性：🟢 日常生活场景 + 现代插画风格 = 天然匹配
- 核心情绪：压力、小心翼翼、人际紧张
- 设计场景：一个年轻人低头看手机，犹豫着要不要发出那条消息
- 映射到风格变量：人物姿态=低头犹豫看手机、服装=日常卫衣、氛围=沉思

**示例（低兼容 🔴，需情绪转译）**：
- 文章主题：「过年回家，亲戚的问话让人窒息」
- 风格：西方古典油画壁画
- 兼容性：🔴 中式春节场景 ≠ 西方古典视觉世界
- 情绪内核：被围观审视、压迫、无处可逃
- 转译场景：一个年轻人站在巴洛克式宴会大厅中央，周围衣着华丽的宾客投来探究的目光，水晶吊灯的光芒既辉煌又令人不安
- 映射到风格变量：人物姿态=孤立站立、环境=华丽但压迫的宴会厅、氛围=辉煌中的不安

#### 正文配图场景设计

**A. 场景图（案例/观点/故事类内容）**

```
1. 提取该段落的核心事件和情绪
2. 执行风格-内容调和（步骤 3.5.0），确定兼容性级别
3. 根据兼容性设计视觉画面：
   - 🟢 → 直接将文章事件可视化
   - 🟡 → 保留事件骨架，替换不兼容的文化/时代细节
   - 🔴 → 提取情绪，在风格视觉世界中找到等价场景
4. 人物、场景是主体
5. 使用风格提示词的共性元素（线条、配色、风格）确保统一
6. 替换风格提示词的可定制变量为设计好的场景
```

**B. 信息图（干货/方法/高密度信息类内容）**

```
1. 提取该段落的关键干货（3-5个要点）
2. 设计知识卡片布局：标题 + 要点列表/图标
3. 关键信息是主体，人物/场景作为点缀
4. ⚠️ 必须保持风格统一：
   - 配色与风格库一致（如蓝色 #0015e4）
   - 线条质感与风格库一致（如手绘线条）
   - 整体视觉风格与场景图协调
5. 信息图的提示词需要明确指定：
   - 卡片布局结构
   - 文字内容（**必须使用中文**，除非用户特别要求英文）
   - 图标/装饰元素风格
6. 信息图中的装饰性人物/场景元素也需遵循风格-内容调和原则
```

### 阶段 4: 提示词生成

**步骤 4.1** 读取风格文件（增强版）

从用户指定的风格文件读取：
- 风格的共性元素（线条、配色、渲染风格等）
- 可定制变量（人物、姿态、场景等）
- 尺寸比例要求（如 16:9）
- **【新增】参考图片提取**：
  1. 解析 `## 原图参考` 区块中的 `![[filename]]` wikilinks
  2. 在 `image_output_dir` 和 vault 根目录下搜索该文件名
  3. 读取图片并 base64 编码，缓存供后续所有配图调用使用
  4. 如有多张参考图，全部读取（Gemini 支持最多 14 张）
  5. 如果找不到参考图文件 → 打印警告，回退到纯文本模式（不阻断流程）

**步骤 4.2** 基于场景设计生成提示词

对每个配图，根据阶段 3.5 的场景设计结果：

**A. 封面提示词**
```
风格共性元素（来自风格文件，保持不变）
    +
封面场景设计的视觉要素（人物状态 + 环境 + 动作 + 情绪）
    +
文章整体主题的语境
    =
完整封面提示词
```

**B. 场景图提示词**（案例/观点/故事类段落）
```
风格共性元素（来自风格文件，保持不变）
    +
该段落的场景设计要素（具体事件的视觉画面）
    +
段落内容的语境
    =
完整场景图提示词
```

**C. 信息图提示词**（干货/方法/高密度信息类段落）
```
风格共性元素中的视觉风格（配色、线条质感）
    +
知识卡片布局（标题 + 要点 + 图标）
    +
提取的关键信息内容
    +
装饰性人物/场景元素（作为点缀）
    =
完整信息图提示词
```

**步骤 4.3** 处理风格变量

将场景设计的具体内容替换风格文件中的可定制变量：
- `{人物姿态}` → 场景设计中的具体姿态（如「低头犹豫看手机」）
- `{服装}` → 场景设计中的服装（如「廓形卫衣」）
- `{场景}` → 场景设计中的环境（如「餐厅角落」）
- `{情绪}` → 场景设计中的氛围（如「沉思、犹豫」）

参考 `references/prompt-composition.md`。

### 阶段 5: 批量生成与插入

**步骤 5.1** 生成封面（如果任务包含）

```
提示词 = 封面场景设计 + 风格共性元素
调用 Nanobanana API（必须传入尺寸参数）
保存图片到指定目录，命名为：[文章名]-封面.png
```

**步骤 5.2** 逐个生成配图

```
for 每个配图位置:
    判断类型：场景图 or 信息图
    生成对应类型的完整提示词
    调用 Nanobanana API（必须传入尺寸参数）
    保存图片，命名为：[文章名]-配图-[序号].png
    记录位置标记
```

**⚠️ 步骤 5.3 API 调用规范（硬性要求）**

调用 Gemini API 生成图片时，**必须**通过 `generationConfig.imageConfig.aspectRatio` 参数指定尺寸比例，不能仅在文字提示词中描述比例（文字描述不可靠）。

**有参考图时（增强模式）**：使用 text + image 混合输入，将风格文件中的原图参考一并传给 API：

```json
{
  "contents": [{"parts": [
    {"text": "Generate an image in this exact visual style. [完整提示词]"},
    {"inline_data": {"mime_type": "image/png", "data": "参考图1-base64"}},
    {"inline_data": {"mime_type": "image/png", "data": "参考图2-base64"}}
  ]}],
  "generationConfig": {
    "responseModalities": ["IMAGE"],
    "imageConfig": {
      "aspectRatio": "16:9"
    }
  }
}
```

**无参考图时（纯文本模式）**：回退到纯文本请求（与原有行为一致）：

```json
{
  "contents": [{"parts": [{"text": "完整提示词"}]}],
  "generationConfig": {
    "responseModalities": ["IMAGE"],
    "imageConfig": {
      "aspectRatio": "16:9"
    }
  }
}
```

**注意：**
- 参考图作为低保真度风格参考（非高保真主体复刻）
- 同一篇文章的所有配图共用相同的参考图（只解析一次）
- 如果风格文件中没有参考图 → 回退到纯文本模式

**尺寸来源优先级：**
1. 风格文件中指定的比例（如风格文件写了 16:9，则使用 16:9）
2. 如风格文件未指定，默认使用 16:9（横版封面）或 1:1（正文配图）

**支持的 aspectRatio 值：** `1:1`, `2:3`, `3:2`, `3:4`, `4:3`, `4:5`, `5:4`, `9:16`, `16:9`, `21:9`

**⚠️ 步骤 5.4 文字语言规范（硬性要求）**

为文章生成的所有图片（封面、场景图、信息图），**图中文字必须使用中文**，除非用户特别要求使用其他语言。包括但不限于：
- 标题、副标题
- 要点/标签文字
- 知识卡片中的信息内容
- 底部标签栏

提示词中涉及图内文字的部分，直接写中文内容交给 API 生成。

**步骤 5.5 文字审校（含文字图片必须执行）**

生成含有文字的图片后，**必须**执行文字审校：

```
1. 读取生成的图片，逐一检查图中所有文字
2. 审校项目：
   - 是否有乱码字符（AI 生图常见问题）
   - 是否有错别字或笔画错误
   - 中文是否被错误替换为英文或其他语言
   - 文字是否清晰可读
3. 判定结果：
   → 文字全部正确 → 通过，继续下一步
   → 发现错误 → 进入修正流程
```

**修正流程：**
```
1. 记录错误文字的位置和内容
2. 保持提示词中其他所有内容不变（构图、风格、人物、配色等）
3. 仅修正错误文字部分
4. 重新调用 API 生成
5. 再次审校，直到文字全部正确
6. 用修正后的图片覆盖原文件
```

> 注意：最多重试 3 次。如果 3 次后仍有文字错误，保留最佳版本并告知用户哪些文字存在问题，由用户决定是否接受或后期手动修改。

**步骤 5.6** 修改 Markdown 文件

在原文件中插入图片：

```markdown
## 某个分小节标题

内容...

![配图描述](相对路径/图片文件名)

继续内容...
```

插入规则：
- 相对路径：从文章所在目录指向图片目录
- 图片描述：使用该分小节的标题作为 alt text
- 保持原有格式：不修改其他内容

**步骤 5.7** 验证与交付

```
显示处理结果：
  ✅ 已生成 X 张配图
  ✅ 已修改文章：[文件路径]

  配图列表：
  - 封面: [文件名]
  - 配图1: [位置] [文件名]
  - 配图2: [位置] [文件名]
  ...

Q: 配图满意吗？
  → 满意 / 需要调整
```

## 内容分析

### 主题类型与配图判断

| 文章类型 | 有配图价值 | 配图位置 | 建议数量 |
|---------|----------|---------|---------|
| **科普** | ✅ 概念/原理/对比 | 讲解关键点后 | 多张 |
| **教程** | ✅ 步骤/结果/演示 | 每个重要步骤后 | 多张 |
| **产品评测** | ✅ 产品/对比/细节 | 评测要点处 | 中等 |
| **旅行** | ✅ 景点/路线/体验 | 各地点介绍处 | 多张 |
| **故事** | ✅ 场景/冲突/人物 | 戏剧化时刻 | 中等 |
| **观点评论** | ⚠️ 仅封面 | 无正文配图 | 仅封面 |
| **新闻** | ⚠️ 仅关键信息处 | 新闻亮点处 | 1-2张 |

### 配图类型判断

| 内容特征 | 配图类型 | 判断依据 |
|---------|---------|---------|
| 讲故事、举例子、描述事件 | **场景图** | 有具体的人物/事件/情境 |
| 表达观点、抒发感受 | **场景图** | 可用氛围画面传达情绪 |
| 列步骤、讲方法、总结要点 | **信息图** | 有可提炼的结构化信息 |
| 展示数据、对比分析 | **信息图** | 有数据或对比关系 |
| 混合型（故事+总结） | **场景图优先** | 以叙事部分为主设计场景 |

### 配图价值评分

对每个分小节评分（1-10），判断是否值得配图：

- **9-10**：强烈推荐配图（具体场景/对象/流程）
- **7-8**：建议配图（有明确主题可视化）
- **5-6**：可配可不配（概念性但可用图补充）
- **1-4**：不推荐配图（纯理论/观点/结论）

详见 `references/content-analysis.md`。

## Pexels 真实照片模式

当用户选择「真实照片」时，使用 Pexels 免费高清照片替代 AI 生成图。适用于所有任务类型（封面/配图/都要）。

### 流程概览

```
文章分析（阶段 2，与风格图模式相同）
    ↓
配图位置判断（阶段 3，与风格图模式相同，但跳过阶段 3.5 场景设计）
    ↓
阶段 P1: 关键词生成
    ↓
阶段 P2: Pexels API 搜索与筛选
    ↓
阶段 P3: 下载并插入
```

> **注意**：阶段 2（文章分析）和阶段 3（配图位置判断）与风格图模式完全一致，复用即可。
> 阶段 3.5（场景设计）、阶段 4（提示词生成）、阶段 5（批量生成与插入）仅适用于风格图模式，真实照片模式跳过。

### 阶段 P1: 关键词生成

**步骤 P1.1** 确定关键词

**用户提供关键词模式：**
- 使用用户在步骤 1.4.5 提供的关键词列表
- 如关键词数量少于配图位置数量 → 询问用户补充或由 AI 补充剩余

**AI 自动提炼模式：**
- 如任务包含封面：基于文章整体主题提取 2-3 个英文搜索关键词（如文章主题「经济下行」→ `economy pressure stress`）
- 对每个「有配图价值」的分小节，提取 2-3 个英文搜索关键词
- 关键词应具体、可视化、适合照片搜索（避免抽象概念）
- **⚠️ 关键词禁止包含人物相关词汇**（如 person、people、man、woman、couple、friends 等），应聚焦于场景、物品、风景、氛围
- 示例：
  - 封面（文章主题「如何选跑鞋」）→ `running shoes close up`
  - 分小节「如何选择跑鞋」→ `running shoes collection`
  - 分小节「朋友请客的承诺」→ `dinner table restaurant food`
  - 分小节「经济下行的焦虑」→ `empty office desk rainy window`

→ 关键词生成后直接进入搜索，不单独确认（减少交互轮次）。

### 阶段 P2: Pexels API 搜索与下载

**步骤 P2.1** API 调用规范

```bash
curl -H "Authorization: [pexels_api_key]" \
  "https://api.pexels.com/v1/search?query=[关键词]&orientation=landscape&per_page=5&page=1"
```

**⚠️ 硬性要求：**
- `orientation` 参数必须设为 `landscape`（横版图片）
- **禁止使用竖版图片**（portrait），确保所有真实照片为横版（16:9 或 4:3 比例）
- 每个关键词搜索 5 张候选图

**步骤 P2.2** 候选图筛选

从 API 返回的 `photos` 数组中，对每张照片提取：
- `id`: 照片 ID
- `photographer`: 摄影师名
- `src.large2x`: 高清下载链接（优先使用）
- `src.large`: 备用下载链接
- `alt`: 图片描述
- `width` / `height`: 确认比例为横版（width > height）

**二次过滤：**
- 排除 width < height 的图片（防止 API 返回竖版）
- **⚠️ 排除含有人类的照片（硬性要求，不可跳过）**：
  - 检查 `alt` 描述，排除含以下关键词的图片：person、people、man、woman、boy、girl、child、baby、couple、portrait、face、selfie、crowd、group、human、worker、employee、model
  - 即使 `alt` 未提及人物，如果从图片 URL 或其他元数据能判断含人 → 也排除
  - 若 5 张候选全部含人 → 更换关键词重新搜索（聚焦物品/场景/风景），最多重试 2 次
- 优先选择与分小节内容语义最相关的图片

**步骤 P2.3** 自动选择最佳图片

对每个配图位置：
1. 根据分小节内容和关键词，从 5 张候选中选择语义最匹配的 1 张
2. 使用 `src.large2x` 下载图片（如不可用，回退到 `src.large`）
3. 下载命令：
```bash
curl -L -o /tmp/[文章名]-配图-[序号].jpeg "[下载链接]"
cp /tmp/[文章名]-配图-[序号].jpeg "[image_output_dir]/[文章名]-配图-[序号].jpeg"
```

> 遵循 iCloud 写入规范：先写 `/tmp` 再 `cp` 到目标路径。

```
  - 配图1: [关键词] → [文件名] (by [摄影师])
  - 配图2: [关键词] → [文件名] (by [摄影师])
  ...

Q: 配图满意吗？
  → 满意 / 需要调整（换图/换关键词）
```

**→ 用户需要调整：**
- 指定哪张图需要替换
- 可提供新关键词或要求从候选图中换一张
- 重新搜索并替换，最多 3 轮

## TMDB 电影剧照模式

当用户选择「电影剧照」时，使用 TMDB (The Movie Database) 电影剧照作为配图。适用于成长/生活/情感类文章，AI 根据文章情绪自动推荐电影并获取剧照，无需用户指定电影名。

### 流程概览

```
文章分析（阶段 2，与其他模式相同）
    ↓
配图位置判断（阶段 3，与其他模式相同，跳过阶段 3.5 场景设计）
    ↓
阶段 T1: 情绪提取与电影匹配
    ↓
阶段 T2: TMDB API 搜索与剧照筛选
    ↓
阶段 T3: 下载并插入
```

> **注意**：阶段 2（文章分析）和阶段 3（配图位置判断）与其他模式完全一致，复用即可。
> 阶段 3.5（场景设计）、阶段 4（提示词生成）、阶段 5（批量生成与插入）仅适用于风格图模式，电影剧照模式跳过。

---

### 阶段 T1: 情绪提取与电影匹配

**步骤 T1.1** 提取文章情绪图谱

读取完整文章后，提取以下信息：

```
1. 主情绪：文章整体传达的核心情感（如孤独、成长、温暖、焦虑、释然）
2. 情绪张力：文章情绪是否有转折？
   - 单一情绪（全程温暖）→ 只需 1 部电影
   - 情绪转折（从焦虑到释然）→ 可考虑 2 部电影
3. 视觉氛围关键词：文章暗示的视觉调性
   - 色调倾向（冷色/暖色/复古）
   - 空间感（封闭/开阔/日常）
   - 时间感（白天/夜晚/黄昏）
```

**步骤 T1.2** AI 推荐候选电影

基于情绪图谱，AI 运用自身电影知识推荐 3-5 部候选电影：

```
选择标准（按权重排列）：
1. 情绪契合度（最高）：电影整体情绪基调是否匹配文章
2. 视觉丰富度：该电影是否以视觉美学著称
3. 剧照可用性：知名电影在 TMDB 上通常有更多高质量剧照
4. 文化相关性：优先选择目标读者群体（中文读者）熟悉的电影
```

> AI 不限于参考表中的电影，可推荐任何合适的电影。参考表见 `references/tmdb-movie-stills.md`。

**步骤 T1.3** 单片 vs 多片决策

```
默认：1 部电影（统一视觉基调）
例外：文章情绪差异大（如前半焦虑后半释然）→ 最多 2 部电影
      每部电影负责不同情绪段落的配图
```

---

### 阶段 T2: TMDB API 搜索与剧照筛选

**步骤 T2.0** TMDB API Key 检查

```
读取 ~/.claude/nanobanana-config.json 中的 tmdb_api_key
  → 存在？ → 继续
  → 不存在？ → 提示用户：
    "电影剧照模式需要 TMDB API Key（免费）。
     请到 https://www.themoviedb.org/settings/api 注册并获取 API Key。"
  → 用户提供 Key → 执行连通性测试 → 保存到 config.json → 继续
```

> 详见 `references/configuration.md` 的「配置项 4: TMDB API Key」。

**步骤 T2.1** 搜索电影获取 ID

```bash
curl "https://api.themoviedb.org/3/search/movie?api_key={key}&query={电影英文名}&language=en-US&page=1"
```

**匹配验证：**
- 搜索结果的 `title` 或 `original_title` 必须与 AI 推荐的电影名一致（忽略大小写和标点差异）
- 如第一条结果不匹配 → 检查前 5 条
- 如全部不匹配 → 用英文原名重试 → 仍不匹配 → 跳过该电影，尝试下一部候选

**步骤 T2.2** 获取剧照

```bash
curl "https://api.themoviedb.org/3/movie/{movie_id}/images?api_key={key}&include_image_language=en,null"
```

**⚠️ `include_image_language=en,null` 是关键参数**，确保获取无文字叠加的纯剧照。

**步骤 T2.3** 筛选排序

对返回的 `backdrops` 数组按以下流水线筛选：

```
1. aspect_ratio >= 1.5 → 只保留横版
2. width >= 1280 → 排除低分辨率
3. 优先 iso_639_1 == null → 无文字叠加的排前面
4. 按质量评分降序：score = vote_average * log2(vote_count + 1)
5. 取前 N 张（封面 top 5，正文每位置 top 3）
```

> 筛选后数量不足的处理见 `references/tmdb-movie-stills.md`。

**步骤 T2.4** 智能分配剧照到配图位置

```
1. 封面：选质量评分最高的剧照
2. 正文配图：根据每个段落的情绪/场景，从候选剧照中选择最匹配的
   - 忧伤段落 → 选冷色调/暗光/空旷场景的剧照
   - 温暖段落 → 选暖色调/室内/人物互动的剧照
   - 紧张段落 → 选强光影对比/逼仄空间的剧照
3. 避免重复：同一张剧照不分配到多个位置
4. 多部电影时：按情绪段落归属分配，保证同一情绪段落用同一部电影的剧照
```

**步骤 T2.5** 呈现匹配结果供用户确认

```
📎 电影剧照匹配结果：

选中电影：Lost in Translation (2003)
选择理由：文章主情绪为"都市孤独感"，该片以东京为背景的孤独与
         人际连接主题高度契合，且以出色的摄影构图著称。

剧照分配计划：
| 位置 | 剧照预览 | 场景描述 |
|------|---------|---------|
| 封面 | [w300 缩略图链接] | 城市夜景霓虹灯 |
| 配图1 (§2) | [w300 缩略图链接] | 独自坐在酒店窗边 |
| 配图2 (§4) | [w300 缩略图链接] | 两人并肩城市街头 |

Q: 确认使用这些剧照？
→ 确认 / 换电影 / 换某张剧照
```

## 信息图模式

当用户选择「信息图」时，AI 分析文章内容，将干货/方法/数据/对比类信息结构化后生成专业信息图。**核心特色：排版优先 + 文字精准度保障 + 基于原图迭代修正。**

详细的布局系统、风格选项、提示词模板和文字审校机制见 `references/infographic-mode.md`。

### 流程概览

```
文章分析（阶段 2，与其他模式相同）
    ↓
阶段 I1: 内容结构化（提取可视化信息）
    ↓
阶段 I2: 布局推荐与风格选择
    ↓
阶段 I3: 提示词生成与图片生成
    ↓
阶段 I4: 文字审校（三级保障）
    ↓
阶段 I5: 插入 Markdown + 交付
    ↓
[可选] 用户反馈 → 基于原图迭代修正
```

---

### 阶段 I1: 内容结构化

**步骤 I1.1** 识别可视化内容

通读文章，提取适合做信息图的段落：
- 列举的要点/方法/步骤（3-8个）
- 对比/对照内容
- 数据/统计
- 分类/层级结构
- 流程/时间线

**步骤 I1.2** 结构化提取

对每个信息点严格从原文摘取（**禁止改写或概括**）：
- 标题/标签（≤8字）
- 正文（原文摘取，≤30字/点）
- 图标关键词（英文，用于匹配图标）
- 层级关系（平级/从属/递进）

**步骤 I1.3** 确定图标题

- 从文章标题或段落标题中提炼，≤15字
- 副标题可选，≤20字
- 单张信息图总文字量 ≤ 200字

---

### 阶段 I2: 布局推荐与风格选择

**步骤 I2.1** 自动推荐布局

根据内容结构自动推荐 2-3 个布局候选（10种布局详见 `references/infographic-mode.md`）：

| 内容特征 | 首选布局 |
|---------|---------|
| 步骤/流程/教程 | flow |
| 对比/选择/优劣 | comparison |
| 多维度/概览/总结 | bento-grid |
| 层级/优先级 | pyramid |
| 时间线/历程 | timeline |
| 核心+分支 | hub-spoke |
| 清单/要点 | checklist |
| 分类/体系 | tree |
| 表象vs本质 | iceberg |
| 转化/筛选 | funnel |

**步骤 I2.2** 风格选择

```
Q: 信息图风格？
→ clean-modern（干净现代，扁平图标）
→ warm-handdrawn（手绘感，暖色调）
→ bold-graphic（大色块，强对比）
→ minimal-mono（极简黑白+强调色）
→ notebook（笔记本风格，荧光笔高亮）
→ morandi-journal（莫兰迪色系，手账感）
→ 复用已有风格（与文章其他配图统一）
```

**步骤 I2.3** 确认

```
显示汇总：
  - 来源：信息图（AI 生成）
  - 任务：[类型]
  - 布局：[推荐布局]
  - 风格：[选择的风格]
  - 文章：[文件路径]
  - 信息图数量：[N 张]
  - 内容预览：[标题 + 要点概要]

Q: 确认无误，开始执行？Y/N
```

---

### 阶段 I3: 提示词生成与图片生成

**步骤 I3.1** 组合提示词

```
布局指令（来自布局定义）
    +
视觉风格指令（来自风格定义）
    +
结构化内容（标题、要点、层级）
    +
文字约束（中文、精准、层级清晰）
    +
排版约束（40-60%留白、一致对齐、图标辅助）
    =
完整信息图提示词
```

提示词模板和布局特定指令详见 `references/infographic-mode.md`。

**步骤 I3.2** 调用 Gemini API

与风格图模式相同的 API 调用规范（步骤 5.3），必须通过 `imageConfig.aspectRatio` 指定尺寸。

信息图默认尺寸：
- 横版（默认）：`3:4` 或 `9:16`（竖版信息图更常见）
- 用户可指定其他比例

保存图片命名：`[文章名]-信息图-[序号].png`

---

### 阶段 I4: 文字审校（三级保障，信息图模式核心）

**第一级：提示词约束**（已在 I3 中嵌入，强制要求文字精准）

**第二级：AI 自审**（每张信息图生成后必须执行）

```
1. 读取生成的图片
2. 逐一核对图中文字与提示词中的原文
3. 检查项：
   a. 字形正确性：每个汉字笔画完整、结构正确
   b. 内容一致性：图中文字与原文完全一致
   c. 数字精确性：所有数字/百分比准确
   d. 排版可读性：文字无重叠、无截断、无模糊
4. 判定：
   → 全部通过 → 进入步骤 I5 交付
   → 发现问题 → 进入第三级修正
```

**第三级：基于原图的迭代修正**

将原图回传 Gemini API，只修改出错的部分，保持其他所有内容不变：

```json
{
  "contents": [{"parts": [
    {"text": "Edit this infographic. Fix ONLY these text errors, keep everything else exactly the same:\n1. Change 'XXX' to 'YYY'\nDo not alter layout, colors, icons, or any other text."},
    {"inline_data": {"mime_type": "image/png", "data": "原图-base64"}}
  ]}],
  "generationConfig": {
    "responseModalities": ["IMAGE"],
    "imageConfig": {"aspectRatio": "原始比例"}
  }
}
```

修正规则：
- 只传达需要改的文字，不重述正确的部分
- 修正后再次执行第二级自审
- 最多自动修正 3 轮
- 3 轮后仍有问题 → 保留最佳版本，列出剩余问题交付用户

---

### 阶段 I5: 插入 Markdown 与交付

插入规则与其他模式相同（见步骤 5.6）。

```
显示处理结果：
  ✅ 已生成 X 张信息图（文字审校通过）
  ✅ 已修改文章：[文件路径]

  信息图列表：
  - 信息图1: [布局] [风格] [文件名]
    内容：[标题]（[N个要点]）
  ...

Q: 信息图满意吗？
  → 满意
  → 文字有误（指出哪里错了）
  → 想改文字（指出要改什么）
  → 想调整布局/风格（重新生成）
```

**用户反馈修正：**

当用户发现文字问题或想修改文字时，使用「基于原图的迭代修正」：
1. 读取当前版本的图片
2. 构建修正提示词（仅描述需要改的部分）
3. 将原图 + 修正指令发送给 Gemini API
4. 生成修正版 → 自审 → 覆盖原文件
5. 最多 3 轮调整

---

### 独立信息图模式

当用户不为文章配图，而是直接要求「做一张信息图」时（用户直接提供内容而非文章文件）：

```
1. 接收用户提供的内容（文字/要点/数据）
2. AI 推荐 2-3 个布局候选
3. 用户选择布局和风格
4. 结构化内容 → 生成提示词 → 调用 API
5. 文字自审 → 修正 → 交付
6. 接受迭代反馈
```

### 改进方向

1. **主题识别准确性**：用户反馈文章内容类型判断是否准确
2. **配图位置智能度**：记录用户是否手动调整过配图位置
3. **风格应用一致性**：生成的图片是否与指定风格一致

### 反馈收集

当用户完成配图后：
- 记录本次使用的文章类型
- 记录生成的配图数量
- 记录用户是否进行了调整
- 如有调整，记录调整内容
